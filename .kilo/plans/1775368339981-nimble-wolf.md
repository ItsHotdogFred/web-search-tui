# Fix Markdown Help Rendering in OpenTUI

## What I found

- `src/info/helpinfo.ts` is the only place using `MarkdownRenderable` and Tree-sitter setup.
- Current parser assets are configured with:
  - `wasm`: `tree-sitter/tree-sitter-markdown` release `v0.3.2`
  - query: `tree-sitter-markdown/main/queries/highlights.scm`
- OpenTUI docs now show a different markdown parser source/version pattern (`tree-sitter-grammars/tree-sitter-markdown`), and document injection mapping for markdown code fences.
- `MarkdownRenderable` works without a `treeSitterClient`; Tree-sitter is optional and mainly required for syntax highlighting (especially fenced code blocks).

## Likely failure mode

- The current markdown parser URL/query combination is likely stale/incompatible, causing client initialization or parser load failures before render.
- Because `createHelpMarkdown()` always initializes Tree-sitter and passes `treeSitterClient`, a parser-loading failure can prevent markdown from rendering at all.
- New evidence from runtime screenshot indicates markdown parsing is happening (markers like `#` and triple backticks are concealed), but style captures are not matching expected tokens, so output appears mostly unstyled.

## Additional findings after first implementation

- This project uses `@opentui/core` `^0.1.95`.
- OpenTUI docs for this generation show broader markdown capture keys than currently defined, including:
  - `markup.heading`
  - `markup.heading.1`, `markup.heading.2`
  - `markup.raw`, `markup.raw.block`
  - `markup.bold`, `markup.strong`, `markup.italic`, `markup.quote`, `markup.link`, `markup.link.url`
- Current `syntaxStyle` only defines `markup.heading.1`, `markup.list`, and `markup.raw`, which may miss the token names emitted by the active markdown highlighter path.

## Implementation plan

1. Update `src/info/helpinfo.ts` to use a robust markdown rendering setup aligned with current OpenTUI docs.
   - Keep `MarkdownRenderable` as the rendering primitive.
   - Use valid parser assets and config for markdown.
   - Include markdown injection config so fenced code blocks can be parsed correctly.
   - Expand `SyntaxStyle` capture map to include both generic and specific markdown scopes (`markup.heading` + `markup.heading.1`, `markup.raw` + `markup.raw.block`, etc.) so styling applies regardless of token granularity.

2. Add resilient Tree-sitter initialization flow.
   - Initialize Tree-sitter in a guarded path (`try/catch`).
   - If parser/bootstrap fails, render markdown anyway (without `treeSitterClient`) so help always appears.

3. Avoid repeated global parser registration and repeated client init on each `@help` invocation.
   - Introduce module-level initialization guard/state.
   - Reuse a prepared client instance across calls.

4. Ensure display behavior remains consistent with current app flow.
   - Keep renderable id `readme` so existing escape cleanup (`renderer.root.remove("readme")`) continues to work.
   - Preserve current markdown style/theme unless a required token change is needed for compatibility.
   - Keep `conceal` default behavior unless explicit markers are desired; if needed for debug clarity, temporarily set `conceal: false` to verify parser/token flow and then restore preferred behavior.

5. Verify with non-dev-server checks.
   - Typecheck compile path (no dev server): `bunx tsc --noEmit` (or equivalent project command).
   - Manual runtime validation path:
     - launch app via existing non-watch command,
     - enter `@help`,
     - confirm markdown body appears,
     - confirm fenced code block renders (highlighted if parser loads, plain if fallback path triggers),
     - confirm `Esc` cleanup still removes help renderable.

## Planned code changes (scope)

- Primary: `src/info/helpinfo.ts`
- No functional change required in `src/views/results.ts` unless a small call-site adjustment is necessary for fallback signaling.

## Acceptance criteria

- `@help` reliably shows markdown content every time.
- Markdown render no longer hard-fails when Tree-sitter assets are unavailable.
- Tree-sitter highlighting works when assets load successfully.
- Existing navigation and cleanup behavior is unchanged.
- Markdown text is visibly styled (heading/list/code colors and heading emphasis), not just plain body text.
